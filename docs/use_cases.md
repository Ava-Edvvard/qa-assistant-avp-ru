# Сценарии использования - QA-Assistant

В данном документе приведены подробные пошаговые руководства по использованию системы **QA-Assistant** для двух основных режимов работы: проектирование с нуля (Новый тест-дизайн) и актуализация существующего тест-плана.

---

## Режим 1: Проектирование тест-дизайна с нуля

Этот сценарий используется, когда у вас есть новые текстовые требования, макеты интерфейсов или таблицы параметров, и вам нужно создать полноценное тестовое покрытие с нуля.

### Шаг 1: Подготовка и настройка модели (LLM)
1. Откройте приложение в браузере по адресу `http://localhost:5173/`.
2. В панели **«Настройка нейросети (LLM)»**:
   - По умолчанию выбран ИИ-провайдер **Kaspersky**.
   - Нажмите кнопку **«Перейти к API»** для перехода в Confluence, скопируйте свой API-ключ и вставьте его в соответствующее поле. 
   - Поле Base URL уже по умолчанию содержит адрес: `https://llm.kaspersky-labs.com/v1/`.
   - Нажмите кнопку **«Загрузить модели»**. После успешной загрузки выберите нужную модель из выпадающего списка (по умолчанию `llama-3.3-70B-instruct`).
   - *Примечание*: Если вы хотите использовать настройки по умолчанию вашего сервера или запустить локальный Mock-режим, оставьте поля пустыми.
3. Нажмите на карточку **«Новый тест-дизайн»**.

### Шаг 2: Ввод исходных требований (Этап 1)
1. В широком текстовом поле **«Исходные требования»** введите ваши требования. Например:
   ```text
   Требования к форме регистрации:
   1. Пользователь должен ввести Email, Пароль и Повтор пароля.
   2. Email должен быть валидным (содержать @ и доменную зону).
   3. Пароль должен быть не менее 8 символов и содержать цифру.
   4. При успешной регистрации отправляется код подтверждения на Email.
   ```
2. В поле **«Дополнительная информация»** можно указать специфичные рамки тестирования (например, *"Тестируем только веб-версию, мобильное приложение не рассматривается"*).
3. При необходимости перетащите или прикрепите файлы:
   - Скриншот макета формы (система извлечет текст с картинки через OCR).
   - Таблицу параметров в формате `.xlsx` (система преобразует строки таблицы в текстовый контекст).
4. Нажмите кнопку **«Продолжить»**.

### Шаг 3: Верификация матрицы требований (Этап 2)
1. Перед вами появится таблица сгенерированных атомарных требований с присвоенными ID (например, `RQ-01`, `RQ-02`, `RQ-03`).
2. Отредактируйте требования, если ИИ разделил их неточно:
   - Нажмите **«Изменить»** в строке требования, чтобы отредактировать описание.
   - Нажмите **«Удалить»**, если требование дублируется или нерелевантно.
   - Нажмите кнопку **«Добавить строку»** внизу таблицы для добавления упущенного требования вручную.
3. Нажмите кнопку **«Сгенерировать вопросы»**.

### Шаг 4: Разрешение неопределенностей с ИИ (Этап 3)
1. ИИ проанализирует матрицу требований и по очереди покажет карточки с уточняющими вопросами. В верхней части вы увидите индикатор прогресса (например, *«Вопрос 1 из 3»*).
2. Для каждого вопроса выберите одно из действий:
   - Введите текст уточнения в текстовое поле и нажмите **«Ответить»** (например: *"Да, пароль должен содержать спецсимволы"*).
   - Если вопрос не важен или относится к функционалу, который не разрабатывается в данном спринте, нажмите кнопку **«Не рассматривается»** — система автоматически запишет ответ *«Не рассматривается в рамках требования»* и перейдет к следующему вопросу.
   - Нажмите **«Пропустить»**, чтобы отложить вопрос на конец очереди.
3. После ответа на все вопросы откроется **«Сводка по ответам на вопросы»**:
   - Проверьте ваши ответы.
   - Если хотите переписать ответ, нажмите **«Изменить ответ»** рядом с нужной строкой, отредактируйте его inline и нажмите **«Сохранить»**.
   - Если хотите вернуться к полноценной карточке вопроса, нажмите **«Вернуть к вопросу»** (вопрос исчезнет из сводки и вернется в активную очередь шага).
4. Нажмите кнопку **«Начать тест-анализ»**.

### Шаг 5: Корректировка тест-сценариев (Этап 4)
1. ИИ сформирует список тест-кейсов с предусловиями, шагами, ожидаемыми результатами, приоритетами и покрытием требований.
2. Проверьте тест-кейсы:
   - Разверните каждый кейс, чтобы изучить шаги.
   - Нажмите кнопку редактирования (иконка карандаша), чтобы изменить приоритет (П1/П2/П3), шаги, ожидаемые результаты или дополнить коды покрытия `coverage`.
   - При необходимости нажмите **«Добавить сценарий»** или **«Удалить»**.
3. Нажмите кнопку **«Далее»**.

### Шаг 6: Анализ покрытия и экспорт результатов (Этап 5)
1. Изучите итоговую интерактивную **Матрицу трассируемости**:
   - Строки соответствуют требованиям (`RQ`), столбцы — тест-кейсам (`TC`).
   - Символ `+` означает, что требование покрыто данным сценарием.
   - Столбец «Покрыт кейсами» показывает общее число тестов на каждое требование.
2. Скопируйте готовые тест-кейсы в формате Markdown (с помощью кнопки **«Скопировать Markdown»**) для переноса в вашу TMS (например, TestIT, Jira, Allure TestOps) или сохраните матрицу покрытия.

## Режим 2: Актуализация существующего тест-дизайна

Этот сценарий используется, когда требования изменились, у вас уже есть старый набор тест-кейсов, и вам нужно быстро обновить его, получив отчет о разнице (Diff Report).

### Шаг 1: Настройка модели и выбор режима
1. Запустите приложение и настройте LLM в соответствии с Шагом 1 предыдущего сценария.
2. Выберите режим **«Доработка существующего»**.

### Шаг 2: Ввод изменений и старых тест-кейсов (Этап 1)
1. В поле **«Исходные требования»** введите новые/обновленные требования к системе.
2. В поле **«Старые тест-сценарии»** скопируйте и вставьте ваши существующие тест-кейсы (в любом текстовом формате).
3. Прикрепите новые макеты или таблицы параметров, если они изменились.
4. Нажмите кнопку **«Продолжить»**.

### Шаг 3: Проверка обновленных требований (Этап 2)
1. Система извлечет список требований, учитывая как новые формулировки, так и старый контекст.
2. Отредактируйте матрицу требований (добавьте новые, удалите неактуальные требования).
3. Нажмите **«Сгенерировать вопросы»**.

### Шаг 4: Разрешение уточняющих вопросов (Этап 3)
1. Ответьте на вопросы ИИ, касающиеся нестыковок между старыми тестами и новыми требованиями.
2. Используйте кнопку **«Не рассматривается»** для пропуска нерелевантных вопросов или отредактируйте ответы на экране сводки.
3. Нажмите **«Начать тест-анализ»**.

### Шаг 5: Ревью новых тест-сценариев (Этап 4)
1. ИИ сгенерирует обновленный набор тест-кейсов, который должен заменить старый.
2. Проверьте шаги, ожидаемые результаты и покрытие, проведите необходимые ручные корректировки в карточках тестов.
3. Нажмите **«Далее»**.

### Шаг 6: Анализ отчета об изменениях (Этап 5)
1. На этом этапе ИИ сравнивает текст старых сценариев (из Шага 1) и структуру новых сценариев (из Шага 4).
2. Прочитайте сгенерированный **Отчет о сравнении (Diff Report)** в формате Markdown:
   - В нем будут перечислены новые добавленные тесты (например, связанные с новым функционалом).
   - Будут указаны модифицированные сценарии (где изменились шаги или ожидаемые результаты).
   - Будут перечислены удаленные или устаревшие тесты.
3. Нажмите кнопку **«Далее»**.

### Шаг 7: Экспорт результатов и отчета (Этап 6)
1. Проверьте финальную матрицу трассируемости.
2. Скопируйте обновленные тест-кейсы и отчет о разнице версий для отчета перед командой или загрузки в систему ведения тестов.
